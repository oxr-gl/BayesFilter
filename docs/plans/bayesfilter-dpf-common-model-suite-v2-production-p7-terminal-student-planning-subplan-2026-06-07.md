# DPF Common Model Suite V2 P7 Terminal Student Planning Subplan

metadata_date: 2026-06-07
phase: P7
status: REVIEWED_READY_FOR_PHASE_EXECUTION

## Question

After v2 BF/FF closure and retirement, what is the correct separate plan for
student implementation repetition without treating students, BF, or FF as an
oracle?

## Inputs

- P0 through P6 v2 results and Claude reviews.
- Prior P6 terminal student classification from the v1 campaign.

## Evidence Contract

Primary criterion:

- A new student-adapter planning artifact exists, but no student implementation
  command is run in the v2 BF/FF production-suite program.
- Student evidence boundary is `static_inventory_only`.

Veto diagnostics:

- any student command before P0 through P6 are closed;
- claiming interface-blocked cells are student failures;
- using BF/FF v2 agreement as an oracle;
- weakening v2 fixture/scalar/gradient contracts for student convenience.
- producing derived student metrics, mismatch labels, or failure labels from
  partial surfaces.

Non-claims:

- P7 does not establish student match, mismatch, correctness, failure, or
  scientific validity.

Artifacts:

- student follow-up plan under `docs/plans/`;
- optional adapter inventory JSON under
  `experiments/dpf_implementation/reports/outputs/`.

Required artifact fields:

- `evidence_boundary = static_inventory_only`;
- `student_filter_commands_run = 0`;
- `student_density_commands_run = 0`;
- `student_path_commands_run = 0`;
- `student_gradient_commands_run = 0`;
- `review_round`;
- `open_material_blockers`;
- `repair_amendment_required`;
- `next_allowed_action`.

## Tasks

1. Summarize v2 closed contract surfaces required from any student adapter:
   density, no-resampling path, fixed-ancestor path, fixed-branch gradients,
   dtype, particles, observations, innovations, ancestors, scalar definitions,
   and checksums.
2. Compare that required surface to currently available student adapters
   by static adapter/runner inventory inspection only.
3. Write a separate reviewed student repetition plan if exact adapter work is
   feasible.
4. Do not run any student filter, density, path, gradient, validation, or
   derived-metric command.
5. Preserve non-claims.

## Exit Criteria

- Student repetition is separated from BF/FF v2 closure.
- No student filter command is run.
- No student density, path, gradient, validation, or derived-metric command is
  run.
- Claude review returns PASS or convergence for the planning boundary.

## Stop Conditions

- User asks to run student code before v2 P0 through P6 closure.
- Student adapter work would require weakening the v2 contract.
